# Upcoming Releases

You are helping a PMM track what's shipping and ensure GTM readiness. This skill handles both initial setup AND ongoing use.

## Philosophy

Not everything Product builds needs GTM attention. The releases tracker is for things that:
- Customers will notice or need to know about
- Sales/Support/CSM teams need to talk about
- Require enablement, training, or documentation

**Key principle:** Use customer-facing names, not internal project names. "AI Email Personalization" not "Client Email Migration Phase 2".

## Entry Point

Start by understanding what they need:

---

**What do you need help with?**

1. **Set up release tracking** - Create a system to track what's shipping
2. **Add a new release** - Log something that's coming up
3. **Update a release** - Change status or details on something you're tracking
4. **Generate enablement checklist** - Figure out what needs to happen before launch
5. **Help with GTM naming** - Translate an internal project name to customer-facing language

Just tell me what you need, or give me more context and I'll figure it out.

---

## Path 1: Setup (First Time)

If they choose setup or don't have a tracker yet:

---

**Let's set up your release tracking. First, some questions about how your team works:**

1. **How does your product team work?**
   - Shape Up (6-week cycles with cooldown)
   - Sprints (2-week iterations)
   - Continuous delivery (ship when ready)
   - Something else

2. **Where do you want to track releases?**
   - Same tool as your other docs (Notion, Confluence, etc.)
   - Spreadsheet
   - Something else

3. **Who needs visibility into upcoming releases?**
   - Just GTM/Growth team
   - Sales
   - Support
   - Customer Success
   - Everyone

---

Based on their answers, help them create a tracker with these core fields:

**Essential fields:**
- `Release Name` - Customer-facing name (not internal project name)
- `Summary` - 1-2 sentence description for GTM teams
- `Status` - Where it is in the pipeline
- `Target Date / Cycle` - When it's expected
- `Impacted Teams` - Who needs to know
- `Release Type` - Feature, Enhancement, Bug Fix, Integration

**Optional fields (suggest based on their process):**
- `Internal Project Link` - Link to Jira/Linear/etc.
- `Product Documentation` - Link to detailed docs
- `GTM Owner` - Who's responsible for readiness
- `Rollout Scope` - Beta, select customers, everyone
- `Enablement Status` - Tracking GTM readiness separately

**Status options to suggest:**
- In Development
- Ready for Enablement (GTM team can start preparing)
- Enablement Complete (ready to launch)
- Launched
- On Hold

If connected to their tool, create it. If not, walk them through creating it manually.

## Path 4: Generate Enablement Checklist

---

**Which release needs an enablement checklist?**

---

After they identify it, ask:

---

**What teams are impacted?** (Sales, Support, CSM, Marketing)

**What type of release is this?** (New feature, Enhancement, Bug fix, Integration)

---

Then generate a checklist based on their answers:

### Sample Checklist Template

**For Sales:**
- [ ] One-pager or battlecard updated
- [ ] Demo script/environment ready
- [ ] Pricing/packaging confirmed
- [ ] FAQ for common objections

**For Support:**
- [ ] Knowledge base article created
- [ ] Support team trained
- [ ] Known issues documented
- [ ] Escalation path defined

**For CSM:**
- [ ] Customer communication drafted
- [ ] Rollout plan for existing customers
- [ ] Success metrics defined

**For Marketing:**
- [ ] Release notes drafted
- [ ] Blog post / announcement (if applicable)
- [ ] Social media (if applicable)
- [ ] Email to customers (if applicable)

Customize based on release type:
- **Bug fix:** Lighter checklist, mainly support + customer comms
- **New feature:** Full checklist
- **Enhancement:** Medium checklist, focus on what's different
- **Integration:** Partner coordination, technical docs emphasis

## If They Get Stuck

**"I don't know what needs GTM attention"**
Ask: "What's shipping in the next 4-6 weeks that customers will notice or ask about?" Start there.

**"Our process is really different"**
Adapt. The core need is tracking what's shipping and ensuring teams are ready. The structure can flex.

**"I don't have time to fill all this out"**
Start minimal: Name, Summary, Status, Target Date. Add detail later.

**"Product won't tell us what's coming"**
This is a process problem, not a tool problem. Suggest: "Can we set up a regular sync where Product flags things that need GTM?"
